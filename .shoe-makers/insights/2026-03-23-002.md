# Insight: Archive Consumed State Files for Decision Traceability

## Lens

**Chain of Custody** — the forensic concept that handling history must be documented at every transfer point. Applied to shoe-makers: when state files are consumed (deleted), the decision context they contained is lost.

## Connection

The three-phase cycle (explore → prioritise → execute) produces then consumes state files. When `candidates.md` is deleted by the prioritise elf, all the rejected candidates and their reasoning are gone. When `work-item.md` is consumed by the executor, the specific instructions that guided the implementation are gone.

The system already archives findings (`.shoe-makers/findings/archive/`). It doesn't archive consumed state files.

## Proposal (reworked from original)

The original insight proposed YAML frontmatter provenance headers on every state file. That's over-engineered.

**Simpler version**: archive consumed state files the same way findings are archived.

When the setup script detects that `candidates.md` or `work-item.md` has been consumed (deleted by the elf that processed it), move the file's last-known content to `.shoe-makers/state/archive/` with a timestamp suffix:

```
.shoe-makers/state/archive/
  candidates-2026-03-23T14-30.md
  work-item-2026-03-23T14-35.md
```

This requires no changes to elf prompts — the setup script already handles housekeeping. The archive would be a git-tracked record of what decisions were made and why (since work items include a "Decision Rationale" section).

**What this enables:**
- An explore elf could read the archive to see what kinds of candidates have been chosen before
- A meta-analysis tool could parse archived work items to find patterns in decision-making
- Debugging: if an implementation has a bug, you can find the work item that produced it

**What this does NOT need:**
- No YAML frontmatter on state files
- No changes to elf prompts
- No commit message conventions
- No new types or data structures

## Why

The system already has the right instinct (archiving findings). Extending the pattern to state files is a one-function change in the setup script. The value is proportional to usage — if the archive is never read, it costs nothing but a few small files. If an elf does read it, it gets decision history for free.
