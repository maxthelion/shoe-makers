# Insight: Shift Retrospective as a First-Class Action

## Lens

Pure function agents — each invocation does one thing and exits. The system loops through explore → prioritise → execute, with adversarial review as cross-elf quality control. But there's no introspective loop where the system evaluates its own process, not just its output.

## Connection

The behaviour tree excels at reactive correction (fix tests, fix critiques) and proactive improvement (explore, prioritise, execute). But it has no mechanism for **meta-improvement** — looking at the shift as a whole and asking "are we spending our ticks wisely?"

## Original Proposal

Add a `retrospect` action that fires at the end of a shift.

## Evaluation

The core insight is sound: the system can improve code but can't improve how it improves code. However, the proposal to add a new tree node that fires "near max-ticks" is complex — it requires new state (tick count tracking) and a condition that depends on the shift runner, not just world state.

**A simpler variant that achieves the same goal**: enhance the setup script's shift summary to include process pattern detection. Currently `summarizeShift()` produces mechanical category counts. If it also analyzed:

- Tick distribution (ratio of reactive to proactive ticks)
- Multi-round review loops (same file reviewed 2+ times)
- Empty explore cycles (explore that produced no candidates)
- Time spent in review vs. execution

...and surfaced these as suggestions in the assessment, the explore elf would naturally pick them up as candidates for process improvement. No new tree node needed.

**Decision: Rework.** The core insight (meta-process improvement) is valuable but the implementation should build on existing mechanisms (shift summary → assessment → explore) rather than adding a new tree node.
