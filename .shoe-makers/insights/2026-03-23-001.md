# Insight: Shift Retrospective as a First-Class Action

## Lens

Pure function agents — each invocation does one thing and exits. The system loops through explore → prioritise → execute, with adversarial review as cross-elf quality control. But there's no introspective loop where the system evaluates its own process, not just its output.

## Connection

The behaviour tree excels at reactive correction (fix tests, fix critiques) and proactive improvement (explore, prioritise, execute). But it has no mechanism for **meta-improvement** — looking at the shift as a whole and asking "are we spending our ticks wisely?"

Currently the shift log accumulates entries, but no elf reads it to derive patterns. The shift summary exists but is mechanical (category counts). There's no equivalent of a team retrospective — a deliberate pause to ask: "We spent 6 ticks on review cycles and only 2 on actual work. Why? Could the review cycle be shorter? Are we generating low-quality work that triggers multi-round reviews?"

## Proposal

Add a `retrospect` action that fires at the end of a shift (when max-ticks is approaching). It reads the shift log, counts how ticks were distributed across action types, and writes a finding with process observations:

- What percentage of ticks were reactive vs. proactive?
- Were there multi-round review/fix cycles? What caused them?
- Did any explore cycles produce zero candidates? Why?
- Were insight evaluations too generous or too dismissive?

This is different from explore (which surveys code) and innovate (which connects to external concepts). Retrospect surveys the **process itself** — it's the system's mechanism for self-improvement at the meta level.

## Why

The current system can improve code but can't improve how it improves code. A team that never retrospects repeats the same process failures. A retrospect action would surface patterns like "we always spend 3 ticks on review for every 1 tick of work" and suggest process changes — fewer files per commit, better test coverage before review, etc.

This closes the loop between doing work and doing work better.
