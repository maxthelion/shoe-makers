# Insight: Dead Letter Queue for Failed Actions

## Lens

**Dead Letter Queue (DLQ)** — a messaging pattern where messages that can't be processed are moved to a separate queue for later inspection rather than being silently dropped or retried forever. DLQs prevent poison messages from blocking the main processing pipeline while preserving them for debugging.

## Connection

The shoe-makers system has no equivalent of a DLQ. When an elf fails (crashes, produces bad code, times out), the shift log records an error entry but the failed action's context is lost. The tree re-evaluates next tick and may attempt the same action again — or may route to something else entirely. There's no way to:

1. See all failed actions across shifts
2. Understand *why* they failed (the prompt, the attempted work, the error)
3. Prevent repeated failures of the same kind

The system handles success well (archive findings, archive state files, review clean work) but handles failure poorly (log an error line, move on).

## Proposal

Add a `.shoe-makers/dead-letters/` directory. When the setup script detects that a previous action failed (e.g., tests broke, typecheck failed after an execute action), it writes the failed action's context to a dead letter file:

```markdown
# Dead Letter — 2026-03-23T15:30:00Z

## Action
execute-work-item

## Work Item (archived)
[content of work-item.md that was being executed]

## Error
Tests failed after execution. 3 test files had failures.

## Test Output
[relevant test output]
```

The explore elf could then read dead letters to identify patterns: "3 of the last 5 dead letters are test failures in the same module — there may be a structural issue." The tree could also add a condition: "if dead letters > N, route to a diagnostic action."

## Why

Currently the system is optimistic — it assumes work succeeds and routes based on positive conditions. The dead letter pattern adds a pessimistic channel: when things go wrong, preserve the context for debugging. This is especially valuable for the shoe-makers system because elves don't persist between ticks — without explicit failure context, each new elf starts from scratch with no knowledge of what went wrong before.
