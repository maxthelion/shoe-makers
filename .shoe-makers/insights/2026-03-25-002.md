# Insight: Commit Message as Structured Data

## Lens

**Unknown** — No Wikipedia article was fetched this tick (network unavailable). Drawing from the observation that commit messages in this system serve dual roles: human-readable narrative AND machine-parseable state signals (e.g., `Auto-commit setup housekeeping` triggers filtering logic).

## Connection

The shoe-makers system already uses commit messages as state signals — the `HOUSEKEEPING_PREFIX` constant in `detect-violations.ts` filters commits by matching message prefixes. The new state-file filter uses `git diff-tree` to check file paths. But the system is evolving toward needing richer commit metadata: what type of action produced this commit? Which skill was used? Was this a reactive or proactive action?

Currently, this metadata is scattered: the action type is in `last-action.md`, the skill is in the tree trace, and the commit message is free-text. A future reviewer elf has to reconstruct the context from multiple sources.

## Proposal

Adopt **structured commit message trailers** (git's `Signed-off-by` convention) for shoe-maker commits. After the human-readable message, append structured metadata:

```
Explore: write candidates focused on innovate observability gaps

Action-Type: explore
Skill: explore
Tick-Phase: proactive
```

The setup script would inject these trailers when the elf commits (or the commit message template could include them). Then:
- `getElfChangedFiles` could filter by `Tick-Phase: housekeeping` instead of string-matching commit messages
- The shift summary could parse commit trailers to build richer dashboards
- The reviewer could see at a glance what role produced each commit without reading `last-action.md`
- The permission violation detector could check `Skill:` trailer against the allowed files for that skill

## Why

- Eliminates fragile string matching (`HOUSEKEEPING_PREFIX`, `STATE_PREFIX`) in favor of structured metadata
- Makes the commit log self-documenting — `git log --grep="Action-Type: innovate"` finds all innovation commits
- Enables richer shift summaries without parsing file paths
- Follows git conventions (trailers are well-supported by `git log --format=%(trailers)`)
- The metadata survives across branches if commits are cherry-picked
