# Insight: Diminishing Returns Detector for Repeated Low-Impact Cycles

## Lens

**Diminishing Returns** — an economic principle where each additional unit of input yields progressively less output. In agriculture: the first fertilizer doubles yield, the second adds 20%, the third adds 2%. Recognizing the inflection point prevents wasted resources.

(No Wikipedia article was fetched this tick — using general knowledge.)

## Connection

The shoe-makers system at innovation tier spends most of its ticks in the innovate → evaluate → review cycle. Each cycle produces one insight that is promoted, reworked, or dismissed. But once the low-hanging fruit is picked (e.g., the first insight wired existing infrastructure together, the second extended an existing pattern), subsequent insights become increasingly speculative and less likely to produce implementable work.

The system has no way to detect this diminishing returns pattern. It will keep cycling through innovate → evaluate → review indefinitely, even if the last 5 insights were all dismissed or produced trivial reworks.

## Proposal

Add a **diminishing returns check** to the explore/innovate decision. When the tree reaches the innovation tier, before routing to `innovate`, check the archive:

- Count recent insights (from `.shoe-makers/archive/` or commit history) in the current shift
- If more than N insights have been produced this shift with a high dismiss rate (>50%), route to `explore` instead

This would be a simple condition check in the tree: `too-many-dismissed-insights?` → route to explore instead of innovate. The explore elf would then survey the codebase with fresh eyes rather than forcing another creative collision.

Implementation: add a `countRecentInsightOutcomes()` function that parses commit messages for "dismiss" vs "promote" keywords in the current shift's commits. If the dismiss ratio exceeds a threshold, the tree routes away from innovate.

## Why

The current system can get stuck in an unproductive creative loop — innovating for innovation's sake. A diminishing returns detector would break the cycle by redirecting to explore, which surveys the actual codebase and may find concrete work that the innovation cycle missed. This is the "sometimes you need to stop brainstorming and go look at the code" instinct, made systematic.
