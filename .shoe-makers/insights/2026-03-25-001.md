# Insight: Uncertainty Budgets for Autonomous Decision-Making

## Lens

**Unknown** — The concept of the unknown itself, and specifically how systems handle epistemic uncertainty. In decision theory and robust control, agents operating under uncertainty use *uncertainty budgets* — explicit quantification of how much they don't know — to calibrate the ambition of their actions. High uncertainty → conservative moves. Low uncertainty → bold moves.

## Connection

The shoe-makers behaviour tree currently uses a binary tier system: if health is good and invariants are green, the system enters innovation tier and starts creative work. But there's no graduated sense of *confidence* in the assessment. A codebase with 221 tested invariants and zero gaps looks the same to the tree whether those invariants were verified five minutes ago or five days ago, whether the health score has been stable for weeks or just recovered from a dip.

The system has no concept of *staleness* or *confidence decay* — assessment.json is either current or it isn't. There's no middle ground.

## Proposal

Add an **uncertainty score** to the assessment that decays over time. Each assessment would track not just the current values but a `confidence` field (0.0–1.0) based on:

1. **Freshness**: how recently the assessment was generated
2. **Stability**: whether scores have been trending consistently (stable = high confidence) or oscillating (volatile = low confidence)
3. **Coverage**: ratio of invariants that have been actively tested in recent shifts vs. inherited from older runs

The behaviour tree's `innovationTier` check would then factor in confidence: only enter creative mode when confidence is above a threshold (e.g., 0.8). Below that, the explore phase would prioritize re-verification and health checks before attempting new work.

This prevents a failure mode where the system happily innovates based on stale data while something has quietly broken.

## Why

Currently, if an external change breaks something that the invariants don't cover, the system has no signal to become more cautious. An uncertainty budget would make the system self-aware about the limits of its own knowledge, naturally routing toward verification when confidence is low and toward innovation when confidence is high. This mirrors how effective human developers work — you check the tests before refactoring, not after.
