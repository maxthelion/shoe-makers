# Insight: Batch Review — Reducing Reactive Overhead

## Lens

**Unknown** — No Wikipedia article was fetched this tick (network unavailable). Drawing instead from the pattern observed during this shift: the review cycle overhead that dominated the reactive ratio.

## Connection

During this shift, roughly 80% of actions were reactive — primarily the critique→fix-critique→re-critique loop that fires after every single commit. Each elf commit triggers an adversarial review, which produces a critique, which must be resolved before new work starts. For proactive work like explore→prioritise→execute, this means 6 actions (3 proactive + 3 reviews) to accomplish one unit of work.

This is analogous to the "stop-the-world garbage collection" problem in runtime systems. Every small allocation triggers a full GC pause. The solution in modern GC is to batch: collect many small allocations, then do one sweep. The shoe-makers system could batch reviews similarly.

## Proposal

Introduce a **batch review mode** for the three-phase orchestration cycle. Instead of reviewing each phase individually:

1. The explore elf writes candidates.md — no review needed (it's just a proposal)
2. The prioritise elf writes work-item.md — no review needed (it's just a plan)
3. The execute elf does the work and commits
4. **One** adversarial review covers the entire cycle: explore decisions + prioritise reasoning + execute output

Implementation: the `hasUnreviewedCommits` condition could recognize that state-file-only commits (candidates.md, work-item.md) are "pending orchestration" and defer review until a source code commit is made. The reviewer would then see the full chain of reasoning that led to the code change.

## Why

- Reduces reactive ratio from ~80% to ~50% for proactive work cycles
- Reviewer gets better context: seeing explore→prioritise→execute as a chain rather than three isolated commits
- Aligns review with what matters: the code change, not the intermediate planning artifacts
- The current system reviews candidates.md, which is just a proposal that will be consumed — reviewing it adds overhead without adding safety
