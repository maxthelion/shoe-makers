### Rebuilt invariants checker with per-claim granularity (`src/verify/invariants.ts`)

**Problem**: Old invariants checker mapped 10 wiki pages to 10 source directories. Since code existed for every directory, it reported 0 spec gaps. But many wiki pages describe 10+ distinct behaviours — most unimplemented. The system couldn't find its own work.

**Solution**: Replaced the coarse TOPIC_TO_CODE mapping with a granular CLAIM_EVIDENCE system:
- **50 individual claims** extracted from 10 wiki pages (was: 10 coarse topics)
- Each claim maps to specific source code patterns and test patterns
- Evidence matching uses AND-of-OR groups: all groups must have at least one match
- Source file comments are stripped before matching to avoid false positives
- The CLAIM_EVIDENCE constant is self-excluded to prevent self-referential matches

**Results on real repo**:
- 42 implemented + tested (was: 10 with everything "fine")
- 2 implemented, untested: `AgentResult` type exists but no agent execution tests
- 6 specified-only gaps (was: 0):
  - `verification.adversarial-llm-review` — no adversarial review function
  - `verification.architectural-contract-check` — no contract checking
  - `behaviour-tree.macro-micro-priority` — macro exists, micro (LLM) doesn't
  - `behaviour-tree.llm-prioritiser` — uses heuristics, not LLM API
  - `observability.findings-directory` — assess doesn't read findings
  - `invariants.four-stage-pipeline` — no LLM extraction stages

**Impact on system behaviour**: `bun run tick` now produces 3 prioritised candidates instead of 0. The system will work instead of sleeping.

**Self-referential matching challenge**: The evidence mapping strings in CLAIM_EVIDENCE were matching as source code evidence (e.g., `"adversarialReview"` in a string literal was found by the evidence scanner). Fixed by stripping the CLAIM_EVIDENCE constant from the invariants file content before scanning.
