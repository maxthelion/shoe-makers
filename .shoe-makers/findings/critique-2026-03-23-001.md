### 2. Dead code: skills/verify.ts, skills/prioritise.ts, skills/work.ts (medium severity)

These three modules are only imported from their test files. No production code imports them. They are functional dead code — they compile, pass tests, but are never called. This is the original issue the spec-code-audit finding identified, and it remains unresolved.

### 3. verify.ts uses execSync — architectural violation (medium severity)

`src/skills/verify.ts:80-85` calls `execSync("bun test", ...)`. The wiki spec (`pure-function-agents.md`, `architecture.md`) and CLAUDE.md say agents are pure functions: "input in, files out, no side effects." Running a subprocess is a side effect. The scheduler should handle running tests; the verify skill should receive test results as input.

### 4. Evidence patterns are too loose (low-medium severity)

Many entries in `src/verify/claim-evidence.ts` use patterns so broad they'd match incidentally:

- `"behaviour-tree.sequence-runs-in-order"` test evidence: `[["sequence"]]` — matching the word "sequence" anywhere in any test file is not evidence that sequence ordering is tested.
- `"branching-strategy.one-branch-per-shift"` test evidence: `[["branch"]]` — matching "branch" proves nothing.
- `"architecture.tree-routes-to-actions"` test evidence: `[["action"]]` — far too broad.
- `"observability.timestamped-entries"` source evidence: `[["timeStamp", "timestamp"]]` — any variable named "timestamp" anywhere would match.
- `"verification.tdd-enforcement"` checks for strings like "TDD" or "Write Tests First" in code — this checks for the *words* about TDD, not whether TDD is actually enforced.

These patterns verify that words exist in the codebase, not that the claimed behaviour is implemented. A claim should not go green because a comment or variable name happens to contain the keyword.

### 5. invariants.md modifications need scrutiny (informational)

The elf added three new invariants and a new section (2.5 Creative exploration, 3.5 Project documentation). The instructions say "only humans maintain invariants.md." However, the invariant `308b879` explicitly says "elves suggest new invariants via findings — the prioritise elf can decide whether to add them." This appears to be self-authorizing: the elf added an invariant saying elves can add invariants. The human should verify these additions are wanted.

### 6. wikipedia.ts has no tests (low severity)

`src/creative/wikipedia.ts` exports `fetchRandomArticle()` and `shouldIncludeLens()`. Neither function has tests. `shouldIncludeLens` is trivially testable (it's `Math.random() < frequency`). `fetchRandomArticle` makes HTTP calls, but the null-return-on-failure path and the stub-length filter could be tested with mocks.

### 7. Minor: integration.md references `bun add shoe-makers` (informational)

The integration guide says `bun add shoe-makers` but shoe-makers is not published as a package. This is aspirational documentation for a capability that doesn't exist yet.
