## Issues

### 1. Evidence patterns are very loose (medium severity)

Many of the new claim-evidence.yaml entries use patterns that verify word presence rather than behaviour. Examples:

- `spec.creative-exploration.the-elf-reads-the-codebase-through-the-lens-and-writes-an-in` — test evidence is `[insight]`. This matches any occurrence of the word "insight" in any test file, not that an elf actually writes an insight file during explore.
- `spec.creative-exploration.insights-are-reviewed-by-a-future-prioritise-elf-who-decides` — test evidence `[insight, prioritise]` just checks those words appear in tests, not that promote/defer/dismiss logic actually works.
- `spec.project-documentation.the-changelog-tracks-userfacing-changes-in-keep-a-changelog-` — source AND test evidence are both `[PROTOCOL_CONTENT]`, which just checks that a constant exists, not that a CHANGELOG is maintained.
- `spec.maintain-a-living-spec.when-wiki-and-code-diverge-the-newer-change-wins-if-the-wiki` — evidence is `[never revert the wiki]` matching prompt text, not verifying that the system actually behaves this way.

This is the same pattern critique-001 flagged. Several of these invariants are "resolved" only because prompt text contains the right words, not because the behaviour is implemented and tested.

### 2. fetchRandomArticle makes network calls during setup (low severity)

`setup.ts` calls `fetchRandomArticle()` which hits the Wikipedia API. This adds network latency and a failure mode to the setup path. The function handles errors gracefully (returns null), so it won't crash, but it means `bun run setup` is no longer purely local. The spec says agents are pure functions with no side effects — while setup isn't an agent, the spirit of the architecture favours determinism. Consider fetching during a dedicated step rather than inline in setup.

### 3. shouldIncludeLens is non-deterministic (low severity)

`shouldIncludeLens` uses `Math.random()`, making the explore prompt non-reproducible. The test for "defaults to 0.3" runs 100 iterations and checks both true and false appear — this is a probabilistic test that could theoretically flake (though the probability is vanishingly small at ~2^-100). Not a real bug but worth noting.

### 4. Deleted inbox messages without explicit confirmation (info)

The `move-claim-evidence-to-yaml.md` was addressed (claim-evidence.ts was already migrated to YAML). The `fix-invariants-granularity.md` was addressed by the broader invariants work. Both deletions seem appropriate.

### 5. claim-evidence.ts deletion is clean (positive)

## Overall Assessment
