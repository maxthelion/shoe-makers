**Design decision**: The elf chose proactive archiving (archive before the elf consumes) rather than reactive archiving (detect consumption after the fact). This is a good choice:
- Avoids the gitignore problem (can't retrieve consumed files from git history)
- Avoids the `HEAD~1` fragility noted in critique 172
- Simple and predictable — archive happens at a known point in the setup flow

### Tests

7 tests covering:
- Archive for execute-work-item, prioritise, dead-code ✓
- No archive for non-consuming actions (explore, critique) ✓
- No archive when file doesn't exist ✓
- Timestamp format validation ✓
- Content preservation ✓

### Potential concerns

1. **Proactive archiving means the archive happens even if the elf fails**: if the setup archives work-item.md but the elf crashes before consuming it, the file is archived but not consumed. Next tick, setup archives it again (new timestamp). This creates duplicate archives. Low impact — the archive is append-only and duplicates are harmless, but worth noting.

2. **No archive cleanup**: the archive grows unbounded. At 2 files per three-phase cycle, this is very slow growth, but over months it could accumulate. No cleanup mechanism exists. Low priority.
