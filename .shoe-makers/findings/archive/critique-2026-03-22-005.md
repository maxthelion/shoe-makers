## Detailed review

### Critique prompt step ordering (src/prompts.ts) — Good fix
The old prompt told reviewers to update `last-reviewed-commit` (step 7) before committing (step 8). Since the marker is gitignored, committing after writing the marker would leave the marker pointing to the pre-review commit, causing the review commit itself to trigger another review cycle. The fix correctly reorders: commit first, then update marker. The added note about the file being gitignored is helpful.

### checkUnreviewedCommits fix (src/state/world.ts) — Good fix
The old `world.ts` version silently returned `false` when no marker file existed, meaning the tree would never route to "critique" until a marker was manually created. The fixed version correctly falls back to checking if ANY commits exist, matching the spec requirement that unreviewed commits should be reviewed.

### Deduplication (setup.ts → world.ts) — Clean refactoring
Both `checkUnreviewedCommits` and `countUnresolvedCritiques` were independently implemented in `setup.ts` and `world.ts`. The refactoring correctly exports from `world.ts` and imports in `setup.ts`. No behavioural change beyond the bug fix.

### Test side-effect fix (world.test.ts) — Important fix
Tests were writing to the real repo's `last-reviewed-commit` marker without restoring it. The beforeEach/afterEach save/restore pattern is the correct fix.

### New invariants tests — Advisory concern

The `parseClaimEvidenceYaml` and `extractInvariantClaims` tests are good unit tests that exercise the YAML parser thoroughly. However:

- **Advisory**: The `extractInvariantClaims` tests use dynamic imports (`await import(...)`) rather than the static import at the top of the file. This is inconsistent with the `parseClaimEvidenceYaml` tests which use the static import. Not blocking, but the inconsistency is worth noting for future cleanup.

### Shell injection surface (advisory)

`git log ${lastReviewed}..HEAD` in `world.ts:64` interpolates the marker file content directly into a shell command via `execSync`. Since the marker is only written by the system itself (commit hashes), this is acceptable. But if the marker file were ever writable by external input, this would be a command injection vector. Worth noting for future reference.
