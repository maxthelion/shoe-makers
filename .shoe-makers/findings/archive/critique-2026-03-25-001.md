Adding `"node"` to the types array is necessary in this restricted environment because the stub `bun-types` doesn't include Node.js types (the real package does). However, this change may cause issues when `bun-types` is properly installed, since both `bun-types` and `@types/node` would provide Node.js type definitions, potentially causing duplicate declaration conflicts. The elf should have noted this as a workaround that may need reverting.

**Severity:** Low — `skipLibCheck: true` in tsconfig likely mitigates duplicate declaration issues, and the real `bun-types` may re-export `@types/node` anyway.

### 3. `any` type annotations are pragmatic but could be tighter (MINOR)

Using `(c: any[])` and `(call: any[])` silences the errors but loses type safety. A tighter fix would be `(c: unknown[])` or extracting a typed variable. However, for test code working with mock spy calls, `any[]` is acceptable.

This is a reasonable workaround — `preconnect` is a Bun-specific extension to the fetch type that the stub doesn't declare. The cast is confined to test code and doesn't affect production behaviour.
