## Findings

### Moderate: `getFrontmatterField` vulnerable to regex injection

**File**: `src/utils/frontmatter.ts:17`

The `field` parameter is interpolated directly into `new RegExp()` without escaping:
```typescript
const match = frontmatter.match(new RegExp(`^${field}:\\s*(.+)$`, "m"));
```
If a field name contains regex metacharacters (`.`, `[`, `(`, etc.), this will silently match the wrong thing or throw. Currently all callers pass safe names (`title`, `category`, `status`), but this is a latent bug. Fix: escape the field name before interpolation.

### Moderate: Dead sentinel check `"Unknown"` in `buildInnovatePrompt`

**File**: `src/prompts/three-phase.ts:187`

```typescript
const hasArticle = article && article.title !== "Unknown";
```

After the refactor, `buildInnovatePrompt` receives `undefined` when no article is available, making the `"Unknown"` check dead code. The `article &&` short-circuits first. This is misleading — it suggests the sentinel is still active. If a Wikipedia article titled "Unknown" were fetched, it would be incorrectly treated as missing. Should be simplified to `const hasArticle = !!article;`.

### Minor: `parseFrontmatter` normalizes CRLF in frontmatter but not body

**File**: `src/utils/frontmatter.ts:9`

Frontmatter gets `.replace(/\r\n/g, "\n")` but the body is returned as-is. Inconsistent — body consumers may get surprised by `\r` characters in CRLF files.

### Minor: `readPreviousActionType` casts without validation

**File**: `src/verify/detect-violations.ts:19`

`raw as ActionType` is used without checking if the value is a valid action type. If corrupted, it would likely cause overly strict (not overly permissive) behavior, so the fail-safe direction is correct.

### Minor: Import placement in `extract-claims.ts`

**File**: `src/verify/extract-claims.ts:25`

The `parseFrontmatter`/`getFrontmatterField` import is placed after the `Claim` interface rather than at the top with other imports.

### Minor: `isAllHousekeeping` now includes `.shoe-makers/state/`
