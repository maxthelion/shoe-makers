## Advisory Notes

### 1. hasUncommittedChanges error handling regression (low risk)

The old `checkUncommittedChanges` in setup.ts had a try-catch returning `false` on error. The shared `hasUncommittedChanges` from world.ts does not catch errors — if `git status` fails, it throws. This is acceptable because setup.ts already assumes a git repo (runs git commands everywhere without guards), but it's a subtle behaviour change worth noting. If the setup ever runs in a non-git context, this will throw instead of gracefully degrading.

### 2. Schedule tests verify execution but not logic (low value)

The new schedule tests for overnight/daytime schedules only assert `typeof result === "boolean"`. They don't actually test that the time comparison logic is correct because they can't control the real clock. The finding from the previous review session noted the same pattern. To properly test this, the time check would need to accept an injectable `now` parameter. Not blocking — the tests prevent regressions from crashes and type errors, which has some value.

### 3. Good: _cachedEvidence removal

Removing the module-level mutable `_cachedEvidence` from invariants.ts is a clear architectural win. Module-level mutables create hidden coupling and make tests order-dependent. Making `claimEvidence` an explicit parameter follows the project's pure-function ethos. Well done.

### 4. Good: enabledSkills filtering

The `loadSkills` change correctly treats `null` as "load all" and `string[]` as a filter list. The tests cover both paths. The `undefined` case (old callers) also works correctly because `undefined && ...` is falsy, falling through to "load all".
