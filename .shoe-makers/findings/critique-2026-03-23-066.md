## Issues Found

### 1. Loose claim-evidence patterns (medium severity)

Several claim-evidence entries use single-word string matches that could pass spuriously:

- `spec.hierarchy-of-needs-in-prioritisation.impact-is-the-primary-criterion-not-riskavoidance` — test evidence is `[highest impact]`. This same string appears in both the gap-tier AND the no-gap-tier of the prioritise prompt, so the test would pass regardless of which tier is active. It doesn't actually verify that impact is the *primary criterion* vs risk-avoidance; it just checks the words appear.

- `spec.hierarchy-of-needs-in-prioritisation.the-explore-and-prioritise-prompts-receive-invariant-counts-` — test evidence is only `[specifiedOnly]`. But `specifiedOnly` is a *variable name* in the source code, not a string that appears in the generated prompt. The test evidence should verify the actual invariant count appears in the prompt output, not that the code references a variable.

  **Wait — correction**: looking more carefully, the invariant checker likely searches test files for these strings. The test `explore uses specifiedOnly count to determine tier` does check `expect(promptWithGaps).toContain("3 unimplemented spec claim")`, which verifies the count is present. But the claim-evidence YAML points to `[specifiedOnly]` which would match the variable name in the test helper `makeStateWithGaps`, not the actual behavioral assertion. This is a loose pattern — it proves the test file mentions the word, not that it verifies the behavior.

### 2. Tier 1 and Tier 2 collapsed (minor, spec alignment question)

The wiki spec says "three tiers: hygiene → implementation → innovation" but the code collapses Tier 1 (hygiene) and Tier 2 (implementation) into a single `Hygiene / Implementation` tier section. The claim-evidence for Tier 1 and Tier 2 both point to the same combined string. This means the code doesn't actually distinguish between hygiene and implementation work — they're a single bucket.

This may be a deliberate simplification, but the spec describes them as separate tiers with separate behaviors. The evidence claims they're separate ("Tier 1: hygiene", "Tier 2: implementation") but the code treats them identically.

### 3. `formatShiftSummary` test checks string presence, not structure (minor)

The `formatShiftSummary` tests check that output contains strings like `"5"`, `"fix, feature"`, `"balanced"`. The check for `"5"` is extremely loose — any markdown containing the digit 5 would pass. The test for "1 errors" is grammatically wrong (should be "1 error") and the test encodes this bug as expected behavior.

### 4. No test for `formatCodebaseSnapshot` edge cases (minor)

`formatCodebaseSnapshot` accesses `assessment.worstFiles` and `assessment.findings.length` without null guards. If `worstFiles` or `findings` were undefined, it would throw. The type system may prevent this, but there's no test covering the function directly.

### 5. Shift summary written after shift completion but not tested in integration (observation)

The `src/shift.ts` change adds `summarizeShift` + `appendToShiftLog` after the shift loop, but there's no integration test verifying this path. The unit tests for `formatShiftSummary` exist, but the wiring in `shift.ts` is untested. This is acceptable for now since it's a simple two-line integration.
