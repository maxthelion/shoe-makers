## Findings

### 1. Backward-compatibility re-export is unnecessary (minor)

`init-templates.ts` re-exports all 9 skill templates from `init-skill-templates.ts` "for backward compatibility". But the only consumer is `init.ts`, which imports from `./init-templates`. This re-export layer adds indirection for no practical benefit — `init.ts` could import directly from `./init-skill-templates`. There are no external consumers. This isn't a bug, but it's unnecessary complexity.

### 2. Duplicate content: skill template vs skill file (advisory)

The `DEPENDENCY_UPDATE_SKILL` template in `init-skill-templates.ts` is identical to `.shoe-makers/skills/dependency-update.md`. This is by design (init creates the skill files from templates), but the elf also committed the skill file directly. When `bun run setup` / `init()` runs, it skips existing files, so there's no conflict — but the committed skill file is redundant with the template. Future elves modifying the skill definition need to update both places. The same pattern exists for the other new skills (octoclean-fix, bug-fix, dead-code) — the templates were added to init but the actual `.shoe-makers/skills/` files already existed.

### 3. dependency-update moved from "Planned Skills" to main list without clear demarcation (minor)

In `wiki/pages/skills.md`, the `### dependency-update` heading was moved from under "## Planned Skills (not yet implemented)" to the main skills list, and the "Planned Skills" section header was removed entirely. This is correct if the skill is now implemented, but the skill is only a *definition* — there's no code in `src/skills/` that implements a dependency-update skill. The skill prompt exists but there's no matching skill implementation. Moving it out of "planned" is premature. The other skills in the main list (implement, fix-tests, etc.) also appear to be prompt-only, so this may be consistent with how the project defines "implemented" — but it's worth noting.
