## Assessment

### 1. claim-evidence.ts deletion and YAML migration (4a17a86) — Mostly good

**Positive:** Correctly deleted the orphaned TypeScript file and added 21 new evidence entries to claim-evidence.yaml. The invariant ID generation fix (stripping non-alphanumeric chars) is necessary and correct.

**Issue — Evidence patterns are too loose for several claims:**

Many of the new section-2 evidence rules use overly generic patterns that would match incidentally rather than verifying the actual behaviour described by the claim:

- `spec.one-invocation-one-action-one-exit.the-scheduled-task-interval-provides-the-loop-not-the-elf`: source evidence is `[tickInterval]`, test evidence is `[tickInterval]`. This checks that the word `tickInterval` appears in source and test, but doesn't verify that the interval *provides the loop*. Any config reference to tickInterval would pass.

- `spec.the-prioritiser-writes-the-real-prompt.*`: Five claims about the prioritiser all have nearly identical evidence: `source: [prioritise] + one other keyword`, `test: [prioritise]`. Finding the word "prioritise" in a test file proves nothing about whether these specific behaviours exist. The test could be testing anything about prioritisation and all five claims would show as "tested".

- `spec.reactive-conditions-top-of-tree-fixed-priority.these-fire-immediately-with-a-focused-prompt-no-orchestratio`: source checks `[makeConditionAction]` and `[generatePrompt]`, test checks `[fix-tests]` and `[critique]`. The claim is about reactive conditions firing *immediately with a focused prompt, no orchestration*. The evidence doesn't distinguish between a reactive condition that fires immediately vs one that goes through orchestration.

**Severity:** Medium. This isn't cheating — the patterns are reasonable first approximations. But the invariants system's value depends on evidence being specific enough to catch regressions. These patterns would stay green even if the actual behaviours they describe were removed, as long as the words remained somewhere.

### 2. Shift log deduplication (6ce2f11) — Correct with a minor fragility

**Positive:** The feature is well-motivated (prevents log spam), implementation is simple, tests cover both dedup and non-dedup cases.

**Minor concern — Regex fragility:** The regex `\n## [^\n]+ — Tick\n\n([\s\S]*?)(?=\n## |\n*$)` relies on the exact format "## ... — Tick\n\n". If the entry format ever changes (e.g., different header style), dedup silently stops working rather than erroring. This is acceptable since the same module controls both write and read, but worth noting.

**Correctness concern:** The regex uses `\n*$` as an alternative lookahead anchor. If the file ends with multiple newlines, the `\n*$` matches greedily before the content ends, potentially capturing trailing whitespace into the match group. The `.trim()` on line 56 compensates for this, so it works in practice, but the regex logic is slightly misleading.
