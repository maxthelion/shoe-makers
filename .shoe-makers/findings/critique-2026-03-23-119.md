## Specific Observations

### 1. `buildDescription` arc narrative is simplistic (minor)

`shift-summary.ts:148` — The arc narrative assumes that having both reactive and explore traces means "started reactive, then stabilised". But the traces aren't checked in chronological order — if the shift started with explore and ended with reactive (things broke late), the description would still say "started reactive, then stabilised", which is misleading.

**Severity**: Low. The description is informational only and doesn't drive behaviour.

### 2. `conditionFires` only counts `passed: true` — naming is accurate but could confuse

`shift-summary.ts:196-198` — `conditionFires` counts conditions that evaluated to `true` (i.e., the condition "fired" and triggered the skill). This is correct behaviour — a condition that evaluates to `false` means the tree moved on. The naming is technically right ("fire" = "triggered") but could be misread as "how many times this condition was evaluated."

**Severity**: Cosmetic. No bug.

### 3. Plural labels have a grammar issue for some actions

`shift-summary.ts:140-142` — The pluralisation logic appends "s" to the label: `"1 dead code removal"` → `"2 dead code removals"`. This works for most labels but `"review"` → `"2 reviews"` is fine, and `"inbox processing"` → `"2 inbox processings"` is awkward.

**Severity**: Cosmetic. Unlikely to appear in practice since inbox actions are rare.

### 4. Insight lifecycle tests verify path strings, not behaviour (acceptable)

`prompts.test.ts:365-402` — The insight lifecycle tests check that certain strings appear in prompts (e.g., `.shoe-makers/insights/`, `YYYY-MM-DD`). These verify prompt content, not that insights are actually read or written. However, this is the correct testing approach for this codebase — prompts are the interface to the LLM agent, and verifying prompt content is how you test that the agent will receive the right instructions.

**Severity**: Not an issue. The test strategy is sound for the domain.

### 5. `observability.md` `last-modified-by` changed from `user` to `elf` — correct

The wiki spec says elves doing doc-sync should update `last-modified-by`. This is appropriate.

### 6. No test for `buildDescription` with errors + arc narrative combined

There's no test that exercises both the arc narrative ("started reactive, then stabilised") and the error suffix (`(2 errors)`) simultaneously. These are independent string-append operations so they compose correctly, but a combined test would document the expected output format.

**Severity**: Very low.
