# Deduplicate REACTIVE_ACTIONS and PROACTIVE_ACTIONS constants

skill-type: health

## Problem

`REACTIVE_ACTIONS` and `PROACTIVE_ACTIONS` are defined identically in two files:

- `src/log/shift-log-parser.ts` lines 5-8
- `src/log/shift-summary.ts` lines 54-57

Both are `Set<string>` with the same values:
```ts
const REACTIVE_ACTIONS = new Set(["fix-tests", "fix-critique", "critique", "review", "inbox"]);
const PROACTIVE_ACTIONS = new Set(["explore", "prioritise", "execute-work-item", "dead-code", "innovate", "evaluate-insight"]);
```

If a new action type is added (or one is reclassified), it's easy to update one file and forget the other. This would cause the reactive ratio calculation and shift summary categorization to disagree — a subtle bug.

## What to do

1. Create `src/log/action-constants.ts` with:
   - `export const REACTIVE_ACTIONS = new Set([...])`
   - `export const PROACTIVE_ACTIONS = new Set([...])`
   - Keep the JSDoc comments

2. Update `src/log/shift-log-parser.ts`:
   - Replace the local `REACTIVE_ACTIONS` and `PROACTIVE_ACTIONS` definitions with imports from `action-constants.ts`

3. Update `src/log/shift-summary.ts`:
   - Replace the local `REACTIVE_ACTIONS` and `PROACTIVE_ACTIONS` definitions with imports from `action-constants.ts`

4. Add a test in `src/__tests__/action-constants.test.ts`:
   - Verify `REACTIVE_ACTIONS` contains the expected actions
   - Verify `PROACTIVE_ACTIONS` contains the expected actions
   - Verify there's no overlap between the two sets
   - Verify the union covers all known action types (import from the tree or check against `TITLE_TO_ACTION` keys)

## Patterns to follow

- Follow the existing module pattern: single-purpose files with named exports
- Keep the new file minimal — just the two constants and their types
- Use the same `Set<string>` pattern already in use

## What NOT to change

- Do not move `TITLE_TO_ACTION` or `ACTION_TO_CATEGORY` — those have additional structure beyond just reactive/proactive classification
- Do not modify the tree definition or any other modules
- Do not touch `.shoe-makers/invariants.md`
- Do not rename the constants — keep `REACTIVE_ACTIONS` and `PROACTIVE_ACTIONS`

## Tests to write

- Test that both sets contain the expected values
- Test that sets are disjoint (no action in both)
- Optionally test completeness against known action types

## Decision Rationale

Chosen over candidate #2 (unify frontmatter parsing) because:
- Lower risk: extracting constants is mechanical, no logic changes
- Higher leverage: these constants directly affect the behaviour tree's reactive ratio signal, which drives action selection
- Addresses the "high reactive ratio" process signal by preventing future bugs in the metric itself
- Candidate #2 (frontmatter parsing) is a larger refactor with more test surface area
