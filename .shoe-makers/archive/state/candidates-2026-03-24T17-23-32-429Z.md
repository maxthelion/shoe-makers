# Candidates

## 1. Unify frontmatter parsing into a shared utility
**Type**: health
**Impact**: medium
**Reasoning**: Three modules parse YAML frontmatter with different regex patterns: `src/verify/extract-claims.ts:29` uses `/^---\s*\n([\s\S]*?)\n---/`, `src/skills/registry.ts:28` uses `/^---\n([\s\S]*?)\n---\n?([\s\S]*)$/`, and `src/skills/assess.ts:70` uses `/^---[\s\S]*?---/m`. The inconsistency means some handle leading/trailing whitespace differently. A single `parseFrontmatter()` utility in `src/utils/` would eliminate drift risk and simplify future changes. All three patterns extract the same thing — YAML between `---` delimiters.

## 2. Add test coverage for init skill template modules
**Type**: test-coverage
**Impact**: low
**Reasoning**: `src/init-skill-templates-docs.ts` and `src/init-skill-templates-quality.ts` define the skill templates written during `shoemakers init`. While referenced in `init-templates.test.ts`, the actual template content and structure aren't directly validated against what `parseSkillFile()` in the registry expects.

## 3. Remove dead production export: evaluate() in tree/evaluate.ts
**Type**: dead-code
**Impact**: low
**Reasoning**: `evaluate()` in `src/tree/evaluate.ts:24` is only used in tests — production always calls `evaluateWithTrace()`. This adds unnecessary API surface. Could be made private or removed if tests switch to using `evaluateWithTrace()` directly (which returns a superset of information).
