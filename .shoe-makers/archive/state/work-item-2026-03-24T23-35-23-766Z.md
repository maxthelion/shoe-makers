# Make RESOLVED_PATTERN more robust to prevent false-positive critique loops

skill-type: bug-fix

## Context

The `RESOLVED_PATTERN` regex in `src/state/world.ts:103` is:
```
/^## Status\s*\n\s*Resolved\.?\s*$/mi
```

This only matches "Resolved." exactly at the end of a line after "## Status". Variants like "Resolved — no action required." or "Resolved. No issues found." fail to match. This causes resolved critiques to be counted as unresolved by `countUnresolvedCritiques`, triggering unnecessary fix-critique cycles.

Experienced firsthand during this shift: critique-073 used "Resolved — no action required." and was not detected as resolved.

## What to change

Modify `RESOLVED_PATTERN` in `src/state/world.ts` to match "Resolved" as a prefix — accept any text after it on the same line.

Change from:
```typescript
export const RESOLVED_PATTERN = /^## Status\s*\n\s*Resolved\.?\s*$/mi;
```

To:
```typescript
export const RESOLVED_PATTERN = /^## Status\s*\n\s*Resolved\b/mi;
```

The `\b` word boundary ensures "Resolved" is a whole word (won't match "Unresolved"), while allowing any suffix like ". No issues." or " — clean." after it.

## Tests to update

In the test file that tests `RESOLVED_PATTERN` or `countUnresolvedCritiques`, add test cases:
- "Resolved." — should match (existing)
- "Resolved" — should match
- "Resolved — no action required." — should match
- "Resolved. Clean." — should match
- "Unresolved" — should NOT match
