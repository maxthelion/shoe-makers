# Implement Structured Critique Skill — Pre-fill Context in Prompt

## Wiki Spec

From `wiki/pages/structured-skills.md`:

> **write-critique** — adversarial review of previous elf's work
> - Setup provides: diff, last-action.md summary, changed file list, commit range
> - Template defines: filename (auto-numbered), sections, status format regex
> - Elf provides: assessment, issues found, severity

> The elf receives a prompt where all the structure is pre-filled and it only needs to provide the intelligent parts.

The spec says setup should gather the diff and commit range during the setup phase (before the elf starts), then embed it in the critique prompt. Currently the elf has to run git commands itself.

## Current Code

`src/prompts/critique.ts:1-29` — `buildCritiquePrompt(permissionViolations?)` generates a free-form prompt that tells the elf to manually read `last-reviewed-commit`, run `git log`, run `git diff`, etc.

`src/setup.ts:131-134` — Setup already detects permission violations for critique actions but doesn't gather diff/commit-range.

`src/setup/format-action.ts:32-34` — `formatAction` passes `permissionViolations` to `generatePrompt` for critique actions.

`src/prompts/index.ts` — `generatePrompt` dispatches to `buildCritiquePrompt`.

## What to Build

1. **In `src/setup.ts`** (around line 131-134 where permission violations are gathered): When `skill === "critique"`, also gather:
   - The last-reviewed-commit hash (from `.shoe-makers/state/last-reviewed-commit`)
   - The commit log: `git log <hash>..HEAD --oneline`
   - The diff: `git diff <hash>..HEAD` (truncate to ~8000 chars if too long, with a note about truncation)
   - The last-action.md content (already read at line 123 as `previousAction`)
   - The auto-numbered critique filename: `critique-YYYY-MM-DD-NNN.md` where NNN is the next sequence number

2. **Update `src/prompts/critique.ts`** — Change `buildCritiquePrompt` signature to accept a context object:
   ```ts
   interface CritiqueContext {
     commitRange: string;       // e.g. "abc1234..HEAD"
     commitLog: string;         // git log output
     diff: string;              // git diff output
     lastAction: string;        // previous elf's action text
     critiqueFilename: string;  // auto-numbered filename
     permissionViolations?: string[];
   }
   ```
   Build the prompt with pre-filled sections per the structured skills spec:
   - Include the diff and commit log inline
   - Include the last-action summary inline
   - Specify the exact output filename
   - Define the exact output format with `[YOUR JUDGEMENT HERE]` placeholders
   - Include the status format note: `Resolved.` on its own line for clean reviews, omit Status section if blocking issues

3. **Update `src/setup/format-action.ts`** — Pass the critique context through `formatAction` to `generatePrompt`.

4. **Update `src/prompts/index.ts`** — Thread the critique context from `generatePrompt` to `buildCritiquePrompt`.

## Patterns to Follow

- Follow the existing pattern where `permissionViolations` flows from `setup.ts` → `formatAction` → `generatePrompt` → `buildCritiquePrompt`. Add the new context alongside it.
- Keep setup.ts changes minimal — use `execSync` for git commands (already imported/used elsewhere in the codebase).
- For the auto-numbering, read the findings directory and archive to find the highest existing NNN for today's date.

## Decision Rationale

Candidate 1 (structured critique) was chosen over candidates 2 (test file split) and 3 (archive test coverage) because:
- It directly addresses the structured skills spec, which is the most impactful unimplemented feature
- It reduces wasted elf ticks on format compliance and manual git command execution during critique
- The critique action fires frequently (every other tick when code changes), so improvements here compound
- Candidates 2 and 3 are lower-impact code quality improvements that can wait
