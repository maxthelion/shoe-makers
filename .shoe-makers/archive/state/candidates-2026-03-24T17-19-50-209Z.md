# Candidates

## 1. Consolidate HOUSEKEEPING_PATHS and SETUP_HOUSEKEEPING_PATHS
**Type**: health
**Impact**: medium
**Reasoning**: `src/setup.ts:175` defines `HOUSEKEEPING_PATHS` with 4 paths (findings, log, archive, state) while `src/utils/housekeeping.ts:6` defines `SETUP_HOUSEKEEPING_PATHS` with 3 paths (findings, log, archive — missing state). These are used for different but related purposes: `HOUSEKEEPING_PATHS` controls auto-commit scope, `SETUP_HOUSEKEEPING_PATHS` filters elf-vs-setup file changes. The subtle difference (state/ included in one but not the other) is intentional but fragile — if a new housekeeping path is added, both must be updated. Consolidate into `src/utils/housekeeping.ts` with a shared base + explicit extensions where needed.

## 2. Unify frontmatter parsing into a shared utility
**Type**: health
**Impact**: medium
**Reasoning**: Three modules parse YAML frontmatter with different regex patterns: `src/verify/extract-claims.ts:29` uses `/^---\s*\n([\s\S]*?)\n---/`, `src/skills/registry.ts:28` uses `/^---\n([\s\S]*?)\n---\n?([\s\S]*)$/`, and `src/skills/assess.ts:70` uses `/^---[\s\S]*?---/m`. The inconsistency means some handle trailing whitespace and others don't. A single `parseFrontmatter()` utility function would eliminate this drift risk and simplify future changes.

## 3. Remove unused production exports: evaluate() and findSkillForType()
**Type**: dead-code
**Impact**: low
**Reasoning**: `evaluate()` in `src/tree/evaluate.ts:24` is only used in tests — production always calls `evaluateWithTrace()`. `findSkillForType()` in `src/skills/registry.ts:123` is also only used in tests. These exports add surface area without production value. They could be kept as test-only utilities or removed if the tests can use `evaluateWithTrace()` directly.

## 4. Add test coverage for init skill template modules
**Type**: test-coverage
**Impact**: low
**Reasoning**: `src/init-skill-templates-docs.ts` and `src/init-skill-templates-quality.ts` define the skill templates written during `shoemakers init`. While referenced in `init-templates.test.ts`, the actual template content and structure aren't directly validated. These templates are the contract for how skills are defined — if a template has a typo or missing frontmatter field, the skill registry won't parse it correctly.
