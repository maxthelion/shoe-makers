# Consolidate housekeeping path constants

skill-type: health

## Problem

Two similar but different housekeeping path constants exist:

- `src/utils/housekeeping.ts:6` — `SETUP_HOUSEKEEPING_PATHS` (3 paths: findings, log, archive)
- `src/setup.ts:175` — `HOUSEKEEPING_PATHS` (4 paths: findings, log, archive, **state**)

The difference (state/ included in one but not the other) is intentional:
- `SETUP_HOUSEKEEPING_PATHS` is used by `detect-violations.ts` to filter false-positive permission violations — state files written by elves (candidates.md, work-item.md) should NOT be filtered
- `HOUSEKEEPING_PATHS` is used by `setup.ts` for auto-commit scope — state files changed by setup ARE housekeeping

But having them in separate files with no shared base makes it easy to add a path to one and forget the other.

## What to do

1. Move `HOUSEKEEPING_PATHS` from `src/setup.ts` to `src/utils/housekeeping.ts`

2. In `src/utils/housekeeping.ts`, define:
   - `CORE_HOUSEKEEPING_PATHS` — the 3 shared paths (findings, log, archive)
   - `SETUP_HOUSEKEEPING_PATHS` — alias for `CORE_HOUSEKEEPING_PATHS` (used by detect-violations)
   - `ALL_HOUSEKEEPING_PATHS` — `CORE_HOUSEKEEPING_PATHS` plus `.shoe-makers/state/` (used by setup auto-commit)
   - Add a comment explaining why state/ is only in the extended set

3. Update `src/setup.ts`:
   - Import `ALL_HOUSEKEEPING_PATHS` from `../utils/housekeeping`
   - Replace `HOUSEKEEPING_PATHS` with `ALL_HOUSEKEEPING_PATHS` in `isAllHousekeeping()` and `autoCommitHousekeeping()`
   - Remove the local `HOUSEKEEPING_PATHS` constant
   - Keep exporting from setup.ts if tests import it (check `src/__tests__/setup.test.ts`)

4. Update `src/verify/detect-violations.ts`:
   - Change import to use `SETUP_HOUSEKEEPING_PATHS` (which is still the 3-path version)

5. Update tests:
   - If `src/__tests__/setup.test.ts` imports `HOUSEKEEPING_PATHS` from setup.ts, update the import to use the new location
   - Add a test verifying `ALL_HOUSEKEEPING_PATHS` is a superset of `SETUP_HOUSEKEEPING_PATHS`

## Patterns to follow

- Follow the pattern just established with `action-constants.ts` — small, focused constants module
- Keep the existing `housekeeping.ts` file, just expand it

## What NOT to change

- Do not change the auto-commit logic in setup.ts
- Do not change the permission violation detection logic
- Do not modify `.shoe-makers/invariants.md`
- Do not rename `isAllHousekeeping` or `autoCommitHousekeeping` functions

## Tests to write

- Verify `ALL_HOUSEKEEPING_PATHS` includes all paths from `SETUP_HOUSEKEEPING_PATHS`
- Verify `ALL_HOUSEKEEPING_PATHS` includes `.shoe-makers/state/`
- Verify `SETUP_HOUSEKEEPING_PATHS` does NOT include `.shoe-makers/state/`

## Decision Rationale

Chosen over candidate #2 (frontmatter parsing) because:
- Directly parallels the action-constants deduplication just completed — same pattern, same risk
- The subtle difference between the two path sets is a known source of confusion (it caused a false positive loop recently per commit history)
- Lower risk than frontmatter refactor since it's just moving constants, not changing parsing logic
