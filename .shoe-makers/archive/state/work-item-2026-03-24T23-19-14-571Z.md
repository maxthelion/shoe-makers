# Tighten critique and review prompts with falsifiable criteria

skill-type: health

## Context

The critique prompt in `src/prompts/reactive.ts` (lines 27-56, `buildCritiquePrompt`) uses subjective review criteria that cause oscillation between reviewers. The wiki spec `wiki/pages/verification.md` (lines 96-100) defines 5 clear questions, but the actual code prompt uses different, vaguer criteria like "tests that don't actually test what they claim" and "evidence patterns that are too loose."

The review prompt (`buildReviewPrompt`, lines 58-69) is even more terse: just "correctness", "tests", "spec alignment" with no structure.

This shift's 65% reactive ratio is substantially driven by these prompts — different LLM invocations disagree on subjective judgments, creating critique→fix→critique oscillation.

## What to change

### 1. Align `buildCritiquePrompt` with the wiki spec

Replace the ad-hoc checklist (lines 40-46) with the 5 questions from `wiki/pages/verification.md` (lines 96-100):

```
1. Did the elf stay within its permitted files?
2. Does the code correctly implement what was asked?
3. Do the tests actually verify the behaviour, or are they trivial?
4. Were any invariants or evidence patterns modified to game the system?
5. Does the change match the wiki spec?
```

Add a verdict instruction requiring the reviewer to state either "Compliant" or "Non-compliant" with specific line references for any issues. This makes the output falsifiable — the next elf can check whether the cited lines actually have the claimed problem.

### 2. Add structure to `buildReviewPrompt`

The review prompt (for uncommitted changes) should also use explicit criteria rather than vague bullets.

### 3. Add a "clean pass" encouragement

The prompt should explicitly say: "If the work is correct and compliant, write a brief 'Clean' verdict and mark it Resolved. Not every review must find problems." This reduces the bias toward always finding something wrong.

## Files to modify

## Decision Rationale

Candidate #1 was chosen over the others because:
- **Highest impact**: Directly addresses the review oscillation that caused 65% reactive ratio. Other candidates (#2 evidence patterns, #3 lifecycle tests, #4 adaptive routing) treat symptoms; this treats the source — the prompts that instruct reviewers.
- **Most specific**: The wiki already has the correct criteria; the code just doesn't use them. This is an alignment fix, not new design.
- **Lowest risk**: Changing prompt text doesn't affect control flow, state management, or data structures. If the new prompts are worse, they can be reverted trivially.
- Candidate #5 (autoCommitHousekeeping) was deprioritised because `checkUnreviewedCommits` in `src/state/world.ts` already independently handles housekeeping-only diffs via `getElfChangedFiles`, making the bug less impactful.
