# Stamp commits with action type to fix permission violation false positives

skill-type: implement

## Original insight (2026-03-24-001)

The permission violation detector reads `last-action.md` to determine what action produced the commits being reviewed. But `last-action.md` gets overwritten each time setup runs. If an elf runs setup twice (e.g. after committing, then checking status), the intermediate run overwrites the tracking, causing the next critique cycle to compare against the wrong action type. This produced 2 false-positive permission violations in a single shift.

## Improved proposal: commit trailers

**Stamp each elf commit with the action type as a git trailer.** The permission detector reads the action type directly from the commits it's checking, eliminating the shared-state race entirely.

### Changes needed

#### 1. `src/setup.ts` — write action type to a file the elf can read

The setup script already writes `next-action.md`. Add a small companion file `.shoe-makers/state/current-action-type` containing just the action type string (e.g. `execute-work-item`). This is read by the elf to stamp its commits.

Actually, simpler: the setup script already writes the action to `last-action.md` via `saveLastAction()`. Instead of changing how the elf commits (which requires teaching every elf to add trailers), **change `detectPermissionViolations` to extract the action type from the commit message** using a convention.

#### 2. `src/setup.ts` — add action type to shift log entry

In `setup.ts` around line 132, the shift log entry already includes the action title. Add the raw action type too:

```
- Action: ${actionTitle}
- Action-Type: ${skill}
```

#### 3. `src/verify/detect-violations.ts` — extract action type from commits

Instead of reading `last-action.md`, read the commit messages in the range and extract the action type. Each setup run logs `Action-Type: <type>` to the shift log, and the auto-commit housekeeping commits that. The elf's own commits don't need trailers — we can determine which action produced them by looking at the most recent setup housekeeping commit before the elf's commits.

**Simpler approach**: Keep `last-action.md` but snapshot it *before* overwriting. Add a `previous-action-type` file that captures the action type before setup overwrites `last-action.md`:

In setup.ts, before `saveLastAction()`:
```typescript
// Snapshot the current action type before overwriting
const previousAction = await readLastAction(repoRoot);
if (previousAction) {
  const prevType = parseActionTypeFromPrompt(previousAction);
  if (prevType) {
    await writeFile(join(repoRoot, ".shoe-makers", "state", "previous-action-type"), prevType);
  }
}
```

Then in `detect-violations.ts`, read `previous-action-type` instead of parsing `last-action.md`.

### The simplest fix

Actually, the simplest fix is: **move `saveLastAction()` to AFTER `detectPermissionViolations()` but BEFORE writing `next-action.md`**. Wait — it's already in that order. The real issue is that setup runs multiple times per session.

The truly simplest fix: **save the action type alongside `last-reviewed-commit`**. When the reviewer writes `last-reviewed-commit`, also write `last-reviewed-action-type`. The permission detector reads this instead of `last-action.md`. This file is gitignored (like `last-reviewed-commit`), so it persists across setup runs without being overwritten.

### Recommended approach

1. In `detect-violations.ts`: read `.shoe-makers/state/previous-action-type` (a new file)
2. In `setup.ts`: before calling `saveLastAction()`, snapshot the current action type to `previous-action-type`
3. Tests: update `detect-violations.test.ts` to verify the new approach

### Files to modify
- `src/setup.ts` (lines ~128-133) — snapshot previous action type
- `src/verify/detect-violations.ts` (lines 15-18) — read `previous-action-type` instead of parsing `last-action.md`
- `src/__tests__/detect-violations.test.ts` — update tests

### What NOT to change
- `src/verify/permissions.ts` — the permission rules are fine, only the action-type detection is broken
- `.shoe-makers/invariants.md` — off-limits
- The commit message format — don't require elves to add trailers

### Tests to write
- Test that `previous-action-type` contains the correct action type after setup runs
- Test that `detectPermissionViolations` reads from `previous-action-type`
- Test the false-positive scenario: setup overwrites `last-action.md` but `previous-action-type` retains the correct value

## Decision

**Promote.** This is a real bug causing real wasted ticks. The fix is small (2 files, ~10 lines of code) and eliminates a class of false positives.
