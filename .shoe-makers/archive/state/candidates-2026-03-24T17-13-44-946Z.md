# Candidates

## 1. Deduplicate action classification constants (REACTIVE_ACTIONS / PROACTIVE_ACTIONS)
**Type**: health
**Impact**: medium
**Reasoning**: `REACTIVE_ACTIONS` and `PROACTIVE_ACTIONS` are defined identically in both `src/log/shift-summary.ts` (lines 54-57) and `src/log/shift-log-parser.ts` (lines 5-8). If one is updated without the other, process pattern detection and shift summaries will disagree — a subtle bug that could cause the behaviour tree to make wrong decisions (e.g., incorrect reactive ratio). Extract to a shared constants module in `src/log/` or `src/utils/`. This is a root cause of potential future reactive churn: a single-point-of-truth violation.

## 2. Unify frontmatter parsing into a shared utility
**Type**: health
**Impact**: medium
**Reasoning**: Three modules parse YAML frontmatter with slightly different regex patterns: `src/skills/registry.ts` (line 28), `src/verify/extract-claims.ts` (line 29), and `src/skills/assess.ts` (line 70). The inconsistent patterns mean some work with trailing newlines and others don't. A single `parseFrontmatter()` utility in `src/utils/` would eliminate this drift risk. Related wiki page: `wiki/pages/skills.md` (skill loading) and `wiki/pages/invariants.md` (claim extraction).

## 3. Add test coverage for init skill template modules
**Type**: test-coverage
**Impact**: low
**Reasoning**: `src/init-skill-templates-docs.ts` and `src/init-skill-templates-quality.ts` define the skill templates written during `shoemakers init`. While referenced in `init-templates.test.ts`, the actual template content and structure aren't directly validated. These templates are the "contract" for how skills are defined — if a template has a typo or missing section, the skill registry won't parse it correctly. Adding targeted tests would catch template format regressions.

## 4. Extract TITLE_TO_ACTION mapping to a shared constant
**Type**: health
**Impact**: low
**Reasoning**: The `TITLE_TO_ACTION` mapping in `src/log/shift-log-parser.ts` (lines 11-23) maps prompt titles to action names. A similar but different mapping `ACTION_TO_CATEGORY` exists in `src/log/shift-summary.ts`. These are semantically related — they both classify the same set of actions but from different directions. If a new action type is added, both must be updated. Centralizing the canonical action list would make this safer.
