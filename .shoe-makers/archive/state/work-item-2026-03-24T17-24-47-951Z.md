# Unify frontmatter parsing into a shared utility

skill-type: health

## Problem

Three modules parse YAML frontmatter with subtly different regex patterns:

1. `src/verify/extract-claims.ts:29` — `/^---\s*\n([\s\S]*?)\n---/` (allows whitespace after opening `---`)
2. `src/skills/registry.ts:28` — `/^---\n([\s\S]*?)\n---\n?([\s\S]*)$/` (strict, also captures body)
3. `src/skills/assess.ts:70` — `/^---[\s\S]*?---/m` (multiline flag, captures full block including delimiters)

All three extract the same thing — the text between `---` delimiters at the top of a markdown file. The differences are accidental, not intentional.

## What to do

1. Create `src/utils/frontmatter.ts` with:
   ```ts
   /**
    * Parse YAML frontmatter from a markdown file.
    * Returns the frontmatter string (without delimiters) and the body after frontmatter.
    * Returns null if no frontmatter found.
    */
   export function parseFrontmatter(content: string): { frontmatter: string; body: string } | null
   ```
   Use a regex that handles all current cases: `/^---\s*\n([\s\S]*?)\n---\s*\n?([\s\S]*)$/`

2. Also add a helper to extract a field from raw frontmatter:
   ```ts
   export function getFrontmatterField(frontmatter: string, field: string): string | undefined
   ```

3. Update `src/verify/extract-claims.ts`:
   - Replace local `parseFrontmatter()` with import from `../utils/frontmatter`
   - Use `getFrontmatterField()` for title and category extraction

4. Update `src/skills/registry.ts`:
   - Replace inline regex in `parseSkillFile()` with import from `../utils/frontmatter`
   - The field-loop parsing (lines 36-41) can stay since it extracts multiple fields

5. Update `src/skills/assess.ts`:
   - Replace inline regex in `findOpenPlans()` with import from `../../utils/frontmatter`
   - The subsequent `category: plan` and `status:` checks can stay on the raw frontmatter string

6. Write tests in `src/__tests__/frontmatter.test.ts`:
   - Standard frontmatter parsing
   - Frontmatter with whitespace after opening `---`
   - No frontmatter → returns null
   - Empty frontmatter → returns empty string
   - Body extraction (text after frontmatter)
   - `getFrontmatterField` with existing/missing fields

## Patterns to follow

- Follow the pattern from `action-constants.ts` and `housekeeping.ts` — small utility modules
- The function should be pure (no side effects, no file I/O)

## What NOT to change

- Do not change the logic of what each consumer does with the parsed frontmatter
- Do not modify `.shoe-makers/invariants.md`
- Do not change the skill registry's validation logic
- Do not add YAML parsing dependencies — keep using regex (matches existing pattern)

## Decision Rationale

Chosen as the natural next consolidation after action constants and housekeeping paths. All three are the same class of issue: duplicated parsing logic with drift risk. This is the largest of the three (3 consumers vs 2), but still mechanical.
