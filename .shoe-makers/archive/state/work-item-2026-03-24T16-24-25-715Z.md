# Work Item: Surface assessment uncertainties in tree trace

## Origin
Insight `2026-03-24-001` — "Taxonomy of Unknowns"

## Problem
When optional features aren't available (octoclean not installed, bun-types missing), the assessment returns `null`/`undefined` and the tree silently skips conditions. The tree trace shows `✗ tests-failing` but doesn't explain *why* — was the condition genuinely false, or was data missing?

## Simplified approach
Instead of refactoring the entire Assessment type to `Uncertainty<T>`, add an `uncertainties` array to the Assessment that records what couldn't be checked and why. Surface this in the tree trace and shift log.

## Implementation

### 1. Add `uncertainties` to Assessment type (`src/types.ts`)

Add to the Assessment interface:
```typescript
uncertainties?: { field: string; reason: string }[];
```

### 2. Record uncertainties in assess.ts (`src/skills/assess.ts`)

When `runTypecheck` returns `undefined`, add `{ field: "typecheckPass", reason: "missing type definitions (bun-types)" }`.
When `getHealthResult` returns `null` and `isOctocleanInstalled` is false, add `{ field: "healthScore", reason: "octoclean not installed" }`.

Build the uncertainties array after the parallel assessment calls, then include it in the assessment object.

### 3. Surface in tree trace (`src/tree/evaluate.ts`)

Add an optional `note` field to `TraceEntry`:
```typescript
export interface TraceEntry {
  condition: string;
  passed: boolean;
  skill: string;
  note?: string;  // e.g., "2 unknowns: typecheckPass, healthScore"
}
```

In `formatTrace`, append notes to failed conditions:
```
  ✗ tests-failing (2 unknowns: typecheckPass, healthScore)
```

### 4. Surface in shift log (`src/setup.ts`)

In `logAssessment`, if `assessment.uncertainties` is non-empty, log:
```
[setup] Uncertainties: typecheckPass (missing type definitions), healthScore (octoclean not installed)
```

### Tests to write
- `src/__tests__/assess.test.ts`: verify uncertainties array is populated when typecheck/health returns undefined/null
- `src/__tests__/evaluate.test.ts`: verify trace note is populated when assessment has uncertainties
- `src/__tests__/setup.test.ts`: verify logAssessment outputs uncertainties

### Patterns to follow
- Keep Assessment backward-compatible (the field is optional with `?`)
- Don't change existing return types — only add the new field
- The tree evaluator currently doesn't have access to the assessment directly — the uncertainties should be surfaced through the WorldState, similar to how other assessment data flows
