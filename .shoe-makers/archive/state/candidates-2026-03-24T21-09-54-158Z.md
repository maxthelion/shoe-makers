# Candidates

## 1. Add missing config keys to README config example
**Type**: doc-sync
**Impact**: medium
**Reasoning**: The README config example (lines 105-111) shows only 5 keys: `branch-prefix`, `tick-interval`, `wiki-dir`, `assessment-stale-after`, `insight-frequency`. But `src/config/load-config.ts` defines 8 known keys — missing from the README are `max-ticks-per-shift`, `max-innovation-cycles`, and `enabled-skills`. Invariant 5.2 says config is configurable for these values. Invariant 3.5 says README reflects current capabilities. This was identified in prior explore cycles but never actioned. Files: `README.md`, `src/config/load-config.ts`.

## 2. Add tests for prompt generation (reactive.ts and three-phase.ts)
**Type**: test
**Impact**: medium
**Reasoning**: `src/prompts/reactive.ts` and `src/prompts/three-phase.ts` generate the actual prompts that elves receive — they are the core interface between the tree and the agents. While `src/__tests__/prompts.test.ts` tests `generatePrompt()` (the index entry point), it may not cover edge cases in the reactive and three-phase modules directly. These modules format permission violations, wiki summaries, Wikipedia articles, and skill definitions into prompts. Testing them directly would catch regressions in prompt formatting. Files: `src/prompts/reactive.ts`, `src/prompts/three-phase.ts`, `src/__tests__/prompts.test.ts`.

## 3. Add tests for shift-log-parser boundary conditions
**Type**: test
**Impact**: low
**Reasoning**: `computeProcessPatterns()` in `src/log/shift-log-parser.ts` has specific logic for review loop detection (3+ consecutive critique/fix-critique) and reactive ratio calculation. The 64% reactive ratio this shift shows these patterns matter for tree behaviour. Boundary cases — exactly 2 consecutive critiques (not a loop), exactly 3 (is a loop), innovation cycle counting — should be explicitly verified. Files: `src/log/shift-log-parser.ts`, `src/__tests__/shift-log-parser.test.ts`.

## 4. Add test for loadSkills filtering by enabledSkills config
**Type**: test
**Impact**: low
**Reasoning**: `loadSkills()` in `src/skills/registry.ts` accepts an `enabledSkills` parameter to filter which skills are loaded. This filtering behaviour should be tested — what happens with an empty list, a list with unknown skill names, null (all enabled). The `enabledSkills` config is documented in invariant 5.2 but the filtering logic is untested. Files: `src/skills/registry.ts`, `src/__tests__/registry.test.ts`.
