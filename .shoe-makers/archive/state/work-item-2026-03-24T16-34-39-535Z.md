# Work Item: Fix permission violation false positive loop

skill-type: bug-fix

## Problem

The system has a self-reinforcing reactive loop where permission violation findings are repeatedly written and resolved, consuming most of each shift as reactive churn (75% reactive ratio this shift).

**Root cause**: `getElfChangedFiles()` in `src/verify/detect-violations.ts` filters out entire housekeeping commits (by message prefix), but doesn't filter out housekeeping *files* that appear in elf commits. When an elf commits work, the shift log (`.shoe-makers/log/`) and archive files (`.shoe-makers/archive/`, `.shoe-makers/findings/archive/`) may be staged alongside the elf's real changes. These housekeeping files then appear as "elf-changed files" and trigger permission violations.

## Implementation

### Fix `getElfChangedFiles()` in `src/verify/detect-violations.ts`

After collecting changed files from elf commits, filter out files under housekeeping paths. Import `HOUSEKEEPING_PATHS` from `src/setup.ts` (already exported there).

**Current code** (lines 86-100):
```typescript
const changedFiles = new Set<string>();
for (const hash of elfCommitHashes) {
  const files = execSync(...).trim();
  if (files) {
    for (const f of files.split("\n")) {
      if (f.length > 0) changedFiles.add(f);
    }
  }
}
return [...changedFiles];
```

**Change to**:
```typescript
const changedFiles = new Set<string>();
for (const hash of elfCommitHashes) {
  const files = execSync(...).trim();
  if (files) {
    for (const f of files.split("\n")) {
      if (f.length > 0) changedFiles.add(f);
    }
  }
}

// Filter out files under housekeeping paths — these are setup-authored,
// not elf-authored, even when they appear in elf commits
return [...changedFiles].filter(f =>
  !HOUSEKEEPING_PATHS.some(p => f.startsWith(p))
);
```

Import `HOUSEKEEPING_PATHS` from `../setup` at the top of the file.
