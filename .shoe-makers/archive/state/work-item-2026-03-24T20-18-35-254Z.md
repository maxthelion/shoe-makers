# Update observability.md — document process patterns, uncertainties, and expected ratios

skill-type: doc-sync

## What the wiki currently says

`wiki/pages/observability.md` describes three types of output (shift log, findings, suggestions) and mentions the `TraceAnalysis` (reactive/routine/explore classification). But it does NOT mention:

1. **Process patterns** — reactive ratio, review loop count, innovation cycle count
2. **Uncertainties** — fields the assessment couldn't check (e.g. missing bun-types, octoclean not installed)
3. **Expected reactive ratios** — what's healthy vs. concerning

## What the code does

### Process patterns (`src/log/shift-log-parser.ts`)
- `parseShiftLogActions()` extracts action names from shift log entries
- `computeProcessPatterns()` computes:
  - `reactiveRatio`: fraction of ticks that were reactive (fix-tests, fix-critique, critique, review, inbox) vs proactive (explore, prioritise, execute, innovate, evaluate-insight, dead-code)
  - `reviewLoopCount`: sequences of 3+ consecutive critique/fix-critique actions
  - `innovationCycleCount`: number of innovate ticks this shift
- `getShiftProcessPatterns()` reads today's shift log and returns the patterns
- These patterns are stored in the `Assessment.processPatterns` field (`src/types.ts` line 75)

### Uncertainties (`src/skills/assess.ts`)
- The assessment tracks fields that couldn't be checked and why
- Stored in `Assessment.uncertainties` as `{ field: string; reason: string }[]` (`src/types.ts` line 77)
- Surfaced in the tree trace (e.g. "2 unknowns: typecheckPass, healthScore")
- Surfaced in the shift log by `logAssessment()` in `src/setup.ts`

### How process patterns are used
- The explore prompt includes a "Process signal" section (e.g. "high reactive ratio (63%)")
- The `formatProcessTemperature()` function in `src/prompts/three-phase.ts` generates guidance based on reactive ratio thresholds
- `>0.6`: "high reactive ratio" — look for root causes of churn
- `<0.3`: "low reactive ratio" — focus on high-impact improvements
- Innovation cycles are capped via `config.maxInnovationCycles`
