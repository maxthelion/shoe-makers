# Candidates

## 1. Tighten critique prompt criteria to reduce review oscillation
**Type**: health
**Impact**: high
**Reasoning**: The 65% reactive ratio this shift is caused by review oscillation — subjective critique criteria in `src/prompts/reactive.ts` (lines 32-55, 58-68) cause different LLM invocations to disagree on code quality. The critique prompt asks reviewers to check for "tests that don't actually test what they claim" and "evidence patterns that are too loose" — both subjective. Replace these with explicit, falsifiable pass/fail criteria derived from `wiki/pages/verification.md` (lines 62-69). This directly reduces the tight commit→critique→fix→critique feedback loop that dominates this shift's reactive cycles.

## 2. Strengthen claim-evidence patterns in claim-evidence.yaml
**Type**: test
**Impact**: high
**Reasoning**: Evidence patterns in `.shoe-makers/claim-evidence.yaml` are too broad. For example, `behaviour-tree.selector-tries-children` just checks for words "selector" and "children" existing — not that they're logically connected. Loose patterns let code pass automated verification while failing manual review, creating critique findings that loop back into reactive work. Tightening patterns in the YAML and adding tests that verify no false-positives would reduce the gap between automated and manual verification, cutting the review loop at its source.

## 3. Add integration tests for the review loop lifecycle
**Type**: test
**Impact**: medium
**Reasoning**: There are no tests verifying that the review→critique→fix cycle terminates. `src/__tests__/` has 45 test files covering individual components but nothing tests the loop as a system: does a finding get created, consumed, archived, and the loop stabilise? Without these tests, behavioural regressions in the loop go undetected. Key files: `src/state/world.ts` (`countUnresolvedCritiques`, `checkUnreviewedCommits`), `src/setup.ts` (`autoCommitHousekeeping`), `src/tree/default-tree.ts` (routing). The `RESOLVED_PATTERN` regex at `src/state/world.ts:103` is fragile — my own critique-073 didn't match because the status line said "Resolved — no action required" instead of the exact "Resolved." pattern.

## 4. Make process pattern signals actionable in tree routing
**Type**: implement
**Impact**: medium
**Reasoning**: `src/log/shift-log-parser.ts` computes `reactiveRatio` and `reviewLoopCount` but these signals are only passed through to explore prompts as informational text (`src/prompts/three-phase.ts` lines 9-30). Wiki spec `wiki/pages/behaviour-tree.md` describes the tree as reactive-first but doesn't prohibit adaptive routing.

## 5. Fix autoCommitHousekeeping marker advancement logic
**Type**: bug-fix
**Impact**: medium
**Reasoning**: `src/setup.ts` lines 200-208: the review marker only advances if `previousMarker === parentOfHead`. When setup auto-commits housekeeping but there are already unreviewed elf commits between the old marker and HEAD~1, the marker stays stale. This means housekeeping commits (archive, shift log) trigger unnecessary review cycles — contributing to the reactive ratio. The fix is to check whether the diff between the old marker and HEAD contains ONLY housekeeping changes, not whether the old marker points to the immediate parent.
