# Add innovation cycle cap — route to explore after N consecutive innovation ticks

## Context

From insight 2026-03-23-004 (diminishing returns): the system can get stuck in an innovate → evaluate → review loop when at innovation tier. Each cycle produces insights of decreasing value. The system needs a way to break this loop and redirect to explore.

## What to build

### Simpler approach than the insight proposed

Instead of parsing commit messages for dismiss/promote rates, use the shift log parser (already built in `src/log/shift-log-parser.ts`) to count how many `innovate` and `evaluate-insight` actions have occurred this shift. If the count exceeds a configurable threshold, the tree should route to `explore` instead of `innovate`.

### 1. Add innovation cycle counting

In `src/log/shift-log-parser.ts`, add a function:

```typescript
export function countInnovationCycles(actions: string[]): number {
  return actions.filter(a => a === "innovate").length;
}
```

### 2. Expose innovation cycle count in world state

Add a field to `WorldState` or pass it via the assessment. The simplest approach: add an `innovationCycleCount` field to the assessment's `processPatterns` object.

In `src/log/shift-log-parser.ts`, update `computeProcessPatterns` to also return `innovationCycleCount`.

### 3. Modify the tree condition for innovation tier

In `src/tree/default-tree.ts`, the `innovation-tier` condition currently just checks `isInnovationTier(assessment)`. Add an additional check: the innovation cycle count must be below a threshold (e.g., 3 per shift).

Or simpler: modify `isInnovationTier()` in `src/prompts/helpers.ts` to also accept the innovation cycle count and return false when it exceeds the cap. This keeps the tree unchanged — only the condition function changes.

### 4. Add config option

In `src/types.ts` `Config` interface, add:
```typescript
/** Maximum innovation cycles per shift before routing to explore (default: 3) */
maxInnovationCycles: number;
```

In `src/config/load-config.ts`, add the default (3) and parsing.

### 5. Tests

- Test that `countInnovationCycles` correctly counts innovate actions
- Test that the tree routes to explore (not innovate) when the cap is reached
- Test that the cap is configurable

## What NOT to change

- Do not modify `invariants.md`
- Do not fundamentally restructure the tree — keep it a simple condition check
- Do not remove the innovation tier — just add a cap

## Patterns to follow

- Follow the existing `isInnovationTier()` pattern
- Follow the config pattern in `load-config.ts`
- The tree condition should remain deterministic

## Decision Rationale

This addresses a real pattern observed during this shift: the system spent 4+ consecutive ticks in innovate → evaluate → review without producing high-impact work. The innovation cycle cap is a simple safety valve that redirects to explore, which surveys the actual codebase and may find concrete work.
