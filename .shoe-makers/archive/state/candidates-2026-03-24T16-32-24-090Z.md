# Candidates

## 1. Fix permission violation false positive loop (root cause of reactive churn)
**Type**: bug-fix
**Impact**: high
**Reasoning**: The system has a self-reinforcing reactive loop: `autoCommitHousekeeping` in `src/setup.ts` commits shift log changes, then `detectPermissionViolations` (line 103-113) detects these as elf-authored changes and writes a `critique-*-permission-violation.md` finding. This finding triggers the `unresolved-critiques` condition, which forces fix-critique, which commits a resolution, which triggers another review cycle. The shift log shows this pattern repeating 6+ times per session.

**Root cause**: In `src/setup.ts` lines 103-113, permission violations are detected AND the finding is written regardless of whether the tree picked "critique". The violation finding should only be written when the tree actually selects the critique action. Additionally, `src/verify/detect-violations.ts` `getElfChangedFiles()` doesn't properly filter out commits made by the setup script's own `autoCommitHousekeeping`.

**Fix**: Move the `writePermissionViolationFinding` call to be conditional on `skill === "critique"` at the point where findings are written, or improve `getElfChangedFiles()` to filter setup-authored housekeeping commits more robustly (matching the `autoCommitHousekeeping` commit message prefix).

**Files**: `src/setup.ts`, `src/verify/detect-violations.ts`, `src/verify/violation-findings.ts`

## 2. Exclude setup-authored commits from permission violation detection
**Type**: bug-fix
**Impact**: high
**Reasoning**: The `getElfChangedFiles()` function in `src/verify/detect-violations.ts` filters commits by prefix `"Auto-commit setup housekeeping"`, but commits made by the setup script for archiving state files (via `archiveConsumedStateFiles`) and writing violation findings themselves are not filtered. Any commit authored by the setup script (not by an elf agent) should be excluded from violation detection. This is a complementary fix to candidate #1.

**Files**: `src/verify/detect-violations.ts`

## 3. Add shift-log-parser tests
**Type**: test
**Impact**: medium
**Reasoning**: `src/log/shift-log-parser.ts` (which provides `getShiftProcessPatterns` used in assessment) has no dedicated tests. The process patterns data (reactive ratio, review loop count, innovation cycle count) drives important behaviour — the explore prompt adapts based on reactive ratio, and the innovation tier cap uses innovation cycle count. Untested parsing logic could silently produce wrong values.

**Files**: `src/log/shift-log-parser.ts` (source), `src/__tests__/shift-log-parser.test.ts` (new)

## 4. Add reactive loop detector to process patterns
**Type**: improve
**Impact**: medium
**Reasoning**: The current `processPatterns` tracks reactive ratio and review loop count, but doesn't detect when the *same* condition fires repeatedly (e.g., "unresolved-critiques" firing 5 times in a row). A simple "consecutive same-action count" metric would help the explore elf diagnose stuck loops and propose root-cause fixes. This addresses the "look for root causes of churn" signal in the explore prompt.

**Files**: `src/log/shift-log-parser.ts`, `src/types.ts` (add field to processPatterns), `src/prompts/three-phase.ts` (surface in explore prompt)

## 5. Improve housekeeping auto-commit timing to prevent review churn
**Type**: improve
**Impact**: medium
**Reasoning**: `autoCommitHousekeeping` in `src/setup.ts` is called twice (line 58 and line 139), creating up to two extra commits per setup run. Each commit can trigger the `unreviewed-commits` condition in the next tick, causing unnecessary review cycles. Consolidating the two calls into a single commit at the end of setup would reduce the number of housekeeping commits and break the review churn pattern.

**Files**: `src/setup.ts`
