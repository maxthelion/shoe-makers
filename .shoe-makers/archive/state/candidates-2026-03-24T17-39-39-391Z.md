# Candidates

## 1. Add getFrontmatterField substring safety test
**Type**: test-coverage
**Impact**: low
**Reasoning**: `getFrontmatterField()` uses `^` with multiline mode to anchor field matching to line starts. This correctly prevents `status:` from matching `sub-status:`. But there's no test verifying this behavior. Adding a test with similar field names (e.g., `title` vs `subtitle`) would document this property and prevent regressions if the regex is ever modified.

## 2. Add test for parseFrontmatter with Windows line endings
**Type**: test-coverage
**Impact**: low
**Reasoning**: `parseFrontmatter()` uses `\n` in its regex, which won't match `\r\n` (CRLF) line endings. If a wiki page is saved with Windows line endings, frontmatter parsing would fail silently. Adding a test documenting this limitation (or fixing the regex to handle `\r?\n`) would prevent surprising failures.

## 3. Verify shift summary computeProcessPatterns uses shared constants
**Type**: test-coverage
**Impact**: low
**Reasoning**: After the action-constants consolidation, `shift-summary.ts` imports `REACTIVE_ACTIONS` and `PROACTIVE_ACTIONS` from the shared module. But there's no direct test in `shift-summary.test.ts` that exercises the process patterns computation through the summary pipeline. Adding one would close the loop on the constants consolidation.
