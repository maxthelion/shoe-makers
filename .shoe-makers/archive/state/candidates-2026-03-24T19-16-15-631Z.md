# Candidates

## 1. Add missing tests for evaluate-insight and innovate prompt paths
**Type**: test
**Impact**: medium
**Reasoning**: `buildEvaluateInsightPrompt()` in `src/prompts/three-phase.ts` is completely untested. `buildInnovatePrompt()` has partial coverage — missing tests for undefined article path and wiki summary formatting. These are rare paths (innovation tier is capped at 3 cycles per shift) but when they fire, untested code runs unsupervised. A prompt bug here could cause the elf to misunderstand the task entirely. No `run-skill.ts` tests exist either. Files: `src/prompts/three-phase.ts`, `src/__tests__/prompts.test.ts`.

## 2. Remove dead code modules (verify.ts, prioritise.ts, work.ts)
**Type**: dead-code
**Impact**: medium
**Reasoning**: Three skill modules in `src/skills/` — `verify.ts`, `prioritise.ts`, `work.ts` — exist but are never called from production code. They're leftovers from the old four-tick model. They have tests but no callers. Dead code confuses elves trying to understand the system and adds to the surface area for invariant checking. A previous audit (`findings/archive/spec-code-audit-2026-03-22.md`) already identified these. Files: `src/skills/verify.ts`, `src/skills/prioritise.ts`, `src/skills/work.ts` and their test files.

## 3. Clarify process temperature signal in tree routing
**Type**: implement
**Impact**: medium
**Reasoning**: The system detects when reactive ratio is high (>60%) via `formatProcessTemperature()` in `src/prompts/three-phase.ts` and `ProcessPatterns` in `src/log/shift-summary.ts`, but this signal only appears in the explore prompt — it doesn't affect tree routing. When reactive ratio is 70%, the system should either investigate root causes or escalate to the human. Adding a tree condition like "reactive-ratio-high" that routes to explore instead of continuing routine cycles would address the root cause of churn. Spec: `wiki/pages/behaviour-tree.md` (hierarchy of needs). Files: `src/tree/default-tree.ts`, `src/log/shift-summary.ts`.

## 4. Review vs critique distinction needs documentation
**Type**: doc-sync
**Impact**: low
**Reasoning**: The tree has two adjacent reactive conditions — `unreviewed-commits → critique` and `uncommitted-changes → review` — with nearly identical prompts. The wiki (`wiki/pages/verification.md` lines 20-24) distinguishes them by scope (critique writes findings only, review can also commit), but `wiki/pages/behaviour-tree.md` doesn't explain the distinction clearly. Documenting the difference would reduce confusion. Files: `wiki/pages/behaviour-tree.md`, `wiki/pages/verification.md`.

## 5. Test coverage for config edge cases
**Type**: test
**Impact**: low
**Reasoning**: `src/config/load-config.ts` handles many edge cases (invalid YAML, missing file, unknown keys) that are tested in `src/__tests__/config.test.ts`, but the integration path — how config flows into tree evaluation and skill loading — is untested. If `enabledSkills` filtering breaks, skills silently disappear. Files: `src/config/load-config.ts`, `src/skills/registry.ts`.
