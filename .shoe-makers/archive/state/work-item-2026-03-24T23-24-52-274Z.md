# Strengthen loose claim-evidence patterns in claim-evidence.yaml

skill-type: test

## Context

Evidence patterns in `.shoe-makers/claim-evidence.yaml` are used by `src/verify/invariants.ts` to verify that spec claims have corresponding code. Some patterns are too broad — they match on individual keywords rather than verifying meaningful code structures. This means code can pass automated verification while not actually implementing the claimed behaviour, creating false confidence and downstream critique cycles.

## What to change

Review the evidence patterns in `.shoe-makers/claim-evidence.yaml` and tighten the loosest ones. Focus on patterns where:
- A single common word is the only evidence (e.g. `[children]`, `[selector]`, `[priority]`)
- The pattern would match unrelated code (false positives)

For each pattern that needs tightening:
1. Read the claim it's verifying (the invariant ID maps to a section in `.shoe-makers/invariants.md`)
2. Read the actual source code that implements the claim
3. Write a more specific pattern that matches the real implementation, not just a keyword

**Examples of patterns to tighten:**
- `behaviour-tree.selector-tries-children` source evidence `[children]` — too broad, matches any use of "children". Should reference the actual selector logic like `["selector", "for", "children"]` or a more specific pattern.
- `behaviour-tree.sequence-runs-in-order` source evidence `[children]` — same issue
- `behaviour-tree.tree-order-priority` test evidence `[priority]` — "priority" appears everywhere

**Do NOT change patterns that are already specific** — e.g. `[export function evaluate]` or `["evaluate(defaultTree"]` are fine.

## Verification criteria

- `bun test` passes after changes
- No invariant claims that previously passed now fail (run `bun test src/__tests__/invariants.test.ts` specifically)
- Each tightened pattern still matches the code it should match
- No new false negatives introduced

## Files to modify

- `.shoe-makers/claim-evidence.yaml` — tighten loose patterns

## What NOT to change

- Do not modify `src/verify/invariants.ts` (the verification engine itself)
- Do not modify `.shoe-makers/invariants.md`
- Do not modify wiki pages
- Do not add new claims — only tighten existing patterns
- Do not remove any claims
