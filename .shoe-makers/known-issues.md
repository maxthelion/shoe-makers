# Known Issues

Read this if you encounter problems. These are issues previous elves have hit.

## Branch checkout creates a new branch instead of using existing remote

**Symptom**: `bun run setup` creates a fresh branch from main instead of checking out the existing remote shoemakers branch. Your push then fails or requires a force push.

**Cause**: `git checkout -b shoemakers/YYYY-MM-DD` runs before `origin/shoemakers/YYYY-MM-DD` is fetched, or the checkout falls through to creating from HEAD.

**Fix**: Run `git fetch origin` first. Then `git checkout shoemakers/YYYY-MM-DD` (without `-b`). If that fails, `git checkout -b shoemakers/YYYY-MM-DD origin/shoemakers/YYYY-MM-DD`.

**NEVER force push to fix this.** If you've committed to the wrong branch, cherry-pick your commits onto the correct one.

## Tests fail with "your current branch 'main' does not have any commits yet"

**Symptom**: Some tests that run git commands fail with this error.

**Cause**: Tests create temporary repos with `git init` but some git versions require an initial commit. These are environment-specific failures — not real test failures.

**Fix**: Ignore these. They pass in the cloud environment.

## Health scan tests fail locally

**Symptom**: `health-scan` tests fail with errors about octoclean or lizard not being installed.

**Cause**: Octoclean requires Python's `lizard` package for complexity analysis. Not installed locally.

**Fix**: These are environment-specific. The health scan gracefully returns null when octoclean isn't available.

## npm registry returns 403 in cloud environments

**Symptom**: `bun install` fails with 403 errors on all packages from registry.npmjs.org.

**Cause**: Some cloud environments block or restrict npm registry access.

**Fix**: If bun is already installed globally (`which bun`), the system can still work — `bun run setup` and `bun test` use local source files and don't require additional npm packages at runtime. Skip `bun install` and proceed directly. If dependencies are genuinely missing, the setup script will fail with import errors — at that point, report the issue and stop.

## Typecheck and health score show as "unknown" in cloud environments

**Symptom**: `bun run setup` reports `Uncertainties: typecheckPass (missing type definitions (bun-types)), healthScore (octoclean not installed)`. The behaviour tree trace shows `✗ tests-failing (2 unknowns: typecheckPass, healthScore)`.

**Cause**: When npm registry access is blocked (see above), `bun-types` and `octoclean` can't install. The typecheck (`tsc --noEmit`) fails because it can't find type definitions for Bun APIs. The health scan returns null because octoclean isn't available.

**Impact**: The `tests-failing` tree condition treats unknowns cautiously — it does NOT trigger when typecheckPass or healthScore are unknown (only when they're explicitly `false`). This means the tree correctly skips fix-tests and proceeds to the next condition. The unknowns are logged but don't block work.

**Fix**: No action needed. This is expected in restricted cloud environments. The system works correctly despite the uncertainties. If you need type checking, ensure `bun install` can access the npm registry.

## Patterns and Gotchas

These aren't bugs — they're things that trip up elves and cause wasted reactive cycles.

### Evidence pattern matching

When writing evidence patterns (in `claim-evidence.yaml`), be aware that:
- Markdown bold (`**text**`) in template literals needs the `*` characters escaped in regex patterns
- Claim IDs are auto-generated by slugifying the wiki page title + section heading, truncated to 60 chars — always verify exact IDs by reading `.shoe-makers/invariants.md` rather than guessing
- Evidence patterns match against source code, so patterns must match the actual code syntax, not prose descriptions

### Invariant claim extraction

The claim extraction regex in `src/verify/extract-claims.ts` matches section headings at specific nesting levels. When writing wiki sections:
- `###` headings create claim groups (two-level numbering like 2.6)
- Deeper nesting (2.6.1) is extracted from numbered list items within those sections
- If you need to verify how claims are extracted, check `extractClaims()` in `src/verify/extract-claims.ts`

### Frontmatter parsing

`getFrontmatterField()` in `src/utils/frontmatter.ts` escapes regex metacharacters in field names, but:
- Stick to simple alphanumeric field names (e.g., `title`, `category`, `status`)
- CRLF line endings are normalized to LF in both frontmatter and body by `parseFrontmatter()`
- Empty frontmatter (`---\n\n---`) is valid and returns an empty string
